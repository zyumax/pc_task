### 課題1-1
SOLID原則の各要素を、業務経験1年目のITエンジニアに伝わるように説明してください。これらを守ることで、どのようなメリットがあるのでしょうか？<br>
#### Single Responsibility 単一責任の原則
モジュールを変更する理由はたった一つだけである。モジュールはひとつのアクターに対して単一の責任を持つ。<br>
一つのことに集中をする<br>
メリットは、モジュールの行うことが明確になるので、保守性が向上する
#### Open-Closed オープンクローズドの原則
- モジュールは、拡張に対しては開いていて、変更に対しては閉じていなければならない
- モジュールは、既存のコードを変更せず新しい機能を追加できるようにすべきである
<br>柔軟に拡張できることを心がける<br>
メリットは、仕様変更の際、既存コードを修正せずに拡張がしやすくなる

#### Liskov Substitution リスコフの置換原則
あるクラスを継承したクラスは、継承元と同じ振る舞いをすべきである<br>
親クラスができることは、子クラスでもできないとダメ<br>
メリットは、親クラスを見れば振る舞いがわかる

#### Interface Segregation インターフェイス分離の原則
インターフェースのクライアントに、クライアントが利用しないプロパティ/メソッドへの依存を強制してはならない<br>
インターフェースを作るときは、必要なものだけを実装する(なんでも詰め込まない)<br>
メリットは、インターフェースを見れば何を実装しているかわかるようになる。また、インターフェースを細かく分けることで、保守性が向上する。

#### Dependency Inversion 依存性逆転の原則
上位レベルのモジュールは下位レベルのモジュールに依存してはならない。両方とも抽象に依存すべきである。依存するとは、あるモジュールが別のモジュールの機能を利用することを指す。<br>このとき、利用する側が上位モジュールで、利用される側が下位モジュールである。<br>
お互いに『共通のルール（インターフェース）』に依存する<br>
（使う側）と（使われる側）が、真ん中にある（インターフェース）という共通の規格を守ること<br>
メリットは、疎結合を実現することができる

### 課題1-2
単一責任の原則と、単純にファイルを細かなファイルに分解することには、どのような違いがあるでしょうか？<br>
<br>
単一責任の原則・・・モジュールを変更する理由は、一つにするべきである(1人のアクターに対して責務を持つ)<br>
単純にファイルを分ける・・・ファイルを細かく分けることによって、管理がしやすくなる。ファイル名から何をしているファイルなのかがわかりやすくなる。<br>
ファイルを分けても、単一責任の原則に則ってるとは限らない

### 課題1-3
Open-Closed-Principleの実例を一つ考えて、作成してみてください。
```
interface PricingStrategy {
  getPrice(price: number): number;
}

class RegularPrice implements PricingStrategy {
  getPrice(price: number):number {
    return price;
  } 
}

class TenPercentOffPrice implements PricingStrategy {
  getPrice(price: number):number {
    return price * 0.9;
  } 
}

class TwentyPercentOffPrice implements PricingStrategy {
  getPrice(price: number):number {
    return price * 0.8;
  }
}
```

### 課題1-6
どんな時に依存性の逆転を用いる必要が生じるのでしょうか？<br>
- 呼び出している対象が 外部モジュールや変わりやすいものの場合(詳細の変更がビジネスロジックに波及してしまう。直接呼び出していると、呼び出す詳細側のメソッドが変更になった際に、呼び出しているメソッド名も同時に変更しなければならない)

### 課題1-7-1
デメテルの法則とは何でしょうか？業務経験1年目のITエンジニアに伝わるように説明してください。この法則を守ることで、どのようなメリットがあるのでしょうか？<br><br>
**自分と関係のあるオブジェクトのみを呼び出していい。自分が呼び出したオブジェクトが関連づけているオブジェクトは呼び出してはいけない。**
- 疎結合なコードを書くことができる(密結合を防ぐことができる)
- 疎結合になることでテストが書きやすくなる
